Hector — Full System Review
What Is Hector?
Hector is a full-stack internal operations platform for a multi-brand e-commerce business (RetroFam, Retro vGames, and related brands). It manages the entire lifecycle from order intake to delivery, supply chain, warehouse operations, HR, and analytics — all in one system.

How Many Modules? 28 Modules across 7 Business Domains

1. Order Processing
The heart of the platform. Handles orders from 11 sales channels simultaneously:

Special sub-workflows inside Order Processing:

WFO (Walmart Fulfillment Operations) — Walmart orders use a dedicated scale-integrated workflow. Physical USB scales connect via SerialPort, weight is captured live (wfoScaleEventModel.js), and rates are fetched from ShipStation per box. Has its own WFO Rates Panel with complex carrier logic.
TOF (Temu Order Fulfillment) — Temu uses its own native label system. Temu issues labels directly; Hector handles PDF label rendering and label history tracking.
Dashboard — Order processing analytics dashboard.
2. Order Fulfillment (+ SSOT)
Handles what happens after an order is picked:

Packing — Warehouse staff pack orders, photos captured, weights recorded
Dropship — Third-party dropship orders; tracks OOS (Out of Stock) items, scraper line sync
Shipping — Generates ShipStation labels, manages multi-carrier rate shopping
SSOT (Single Source of Truth) — A pre-computed master collection that aggregates packing + dropship + shipping into one queryable record per order. Updated asynchronously via BullMQ queue (not real-time writes). Lives in its own ssot/ pages.
Performance & Activity Logs — Fulfillment team performance tracking
3. eBay Scrapper
Dedicated module for eBay-sourced orders — separate from regular eBay marketplace:

Manages buyer accounts (multiple eBay buyer accounts)
Generates labels via EasyPost (not ShipStation)
Tracks shipments via EasyPost webhooks
Has its own health check routes

Key Architectural Notes (for whoever receives this)
Monorepo structure: Hector-Main/server (Node.js/Express backend) + Hector-client (React 19 + Vite frontend). They live in the same repo.

Authentication: JWT + express-session. Role-based permissions enforced by RequirePermission on the client and requirePermission() middleware on the server. Every module has a registry key.

Silent printing via QZ Tray is a core warehouse feature — every label (ShipStation, WFO, TOF native labels) is printed without any browser dialog.

SSOT is not a real-time DB write — it's an eventually-consistent pre-computed record updated by BullMQ workers after fulfillment events. Never write to it directly.

Two inventory systems coexist — Legacy (inventory) and New (inventoryManagement). Migration is in progress; new features go only in the new system.

Multi-warehouse from the ground up — Procurement, transfers, inventory, QA stages all scope by warehouse.

Hardware integration is real — DYMO USB scales communicate via Node.js SerialPort directly from the server, with weight telemetry pushed to the browser via Socket.IO for the WFO workflow.

The pricing engine runs competitor scrapes via Cheerio (server-side HTML scraping) against 10+ competitor sites, queued through BullMQ.
